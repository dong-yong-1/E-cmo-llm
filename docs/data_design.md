# 电商客服大模型数据设计文档

## 1. 设计目标

本项目的数据设计目标不是单纯构造问答对，而是为电商客服场景建立一套可训练、可评估、可分析的结构化数据体系。数据设计需要同时服务于以下几个环节：

- 指令微调训练
- 规则准确率评估
- badcase 分类分析
- LoRA 消融实验中的分桶对比

因此，数据结构不仅要包含用户问题和客服回复，还要包含场景标签、规则标签、难度标签和来源标签，保证后续实验可以在统一 schema 下完成闭环分析。

## 2. 设计原则

本项目的数据设计遵循以下原则：

- 结构化优先：避免只有自然语言问答，尽量保留关键上下文字段
- 规则显式化：将平台规则和策略标签写入样本，突出业务约束
- 场景可分桶：支持按一级场景、二级场景、难度进行统计
- 兼容训练与评估：同一份 schema 同时服务 SFT 训练和模型评测

## 3. Schema v1

当前项目统一采用 `schema v1`，在 [scripts/dataset_schema.py](/Users/dongyong/Project/Trea_code/ecom-llm-cs/scripts/dataset_schema.py) 中定义。

每条样本的标准格式如下：

```json
{
  "id": "售后_退货退款_00001",
  "category": "售后",
  "subcategory": "退货退款",
  "instruction": "用户问：已经拆封还能退吗？",
  "input": {
    "product_category": "护肤品",
    "order_status": "已签收",
    "is_opened": true,
    "has_quality_issue": false,
    "user_emotion": "neutral",
    "platform_rule": "拆封后非质量问题通常不支持7天无理由退货。"
  },
  "output": "您好，若商品已拆封且无质量问题，通常不支持7天无理由退货。如您怀疑商品存在异常，也可以提供订单信息和商品照片，我们帮您进一步核实。",
  "policy_tags": ["退货退款", "不可无理由退货"],
  "quality_label": "good",
  "difficulty": "medium",
  "source": "synthetic",
  "schema_version": "ecom_cs_schema_v1"
}
```

## 4. 字段定义

### 4.1 核心字段

- `id`
  - 样本唯一标识，用于 badcase 跟踪和实验复现。
- `category`
  - 一级场景标签，例如 `售前`、`物流`、`售后`、`投诉`。
- `subcategory`
  - 二级场景标签，例如 `退货退款`、`换货补发`、`催发货`、`地址修改`、`商品咨询`、`投诉安抚`。
- `instruction`
  - 用户问题文本，保留自然语言表达。
- `input`
  - 结构化上下文信息，是训练和评估中最关键的业务条件字段。
- `output`
  - 标准客服回复，作为 SFT 的监督目标。

### 4.2 分析字段

- `policy_tags`
  - 样本对应的规则标签，用于规则准确率评估和错误归因。
- `quality_label`
  - 数据质量标记，当前保留 `good`、`needs_review`、`bad` 这类取值空间。
- `difficulty`
  - 样本难度，用于分桶评测。
- `source`
  - 数据来源，例如 `synthetic`、`manual`、`legacy`、`augmented`。
- `schema_version`
  - schema 版本标识，用于保证训练、评估和数据生成逻辑一致。

## 5. Input 字段设计

`input` 采用结构化字典，而不是纯文本，目的是方便模型输入格式标准化、后续规则校验和 badcase 归因。

当前版本固定字段如下：

- `product_category`
  - 商品类别，例如服饰、护肤品、数码配件、家居用品。
- `order_status`
  - 订单状态，例如 `待发货`、`运输中`、`已签收`。
- `is_opened`
  - 商品是否拆封。
- `has_quality_issue`
  - 是否存在质量问题。
- `user_emotion`
  - 用户情绪，例如 `neutral`、`anxious`、`angry`。
- `platform_rule`
  - 当前样本必须遵守的平台规则说明。

训练时会通过统一格式函数将 `input` 序列化为文本，例如：

```text
用户问：已经拆封还能退吗？
商品类别：护肤品
订单状态：已签收
是否拆封：是
是否质量问题：否
用户情绪：neutral
平台规则：拆封后非质量问题通常不支持7天无理由退货。
```

## 6. 场景体系设计

当前数据生成脚本 [data_scripe.py](/Users/dongyong/Project/Trea_code/ecom-llm-cs/data_scripe.py) 中已定义以下一级场景与二级场景：

- `售后 / 退货退款`
- `售后 / 换货补发`
- `物流 / 催发货`
- `物流 / 地址修改`
- `售前 / 商品咨询`
- `投诉 / 投诉安抚`

这套设计覆盖了电商客服中的高频问答和规则敏感场景，尤其强调：

- 退货退款类规则判断
- 发货前后地址修改差异
- 投诉场景的安抚与处理路径
- 商品咨询类问题中的信息不足处理

## 7. Policy Tags 设计

`policy_tags` 是本项目数据设计中的关键字段。它的作用不是辅助展示，而是为了支持后续规则评估。

当前标签设计由两部分组成：

- 二级场景标签
  - 例如 `退货退款`、`地址修改`、`投诉安抚`
- 规则标签
  - 例如 `可无理由退货`、`不可无理由退货`、`质量问题可退`、`发货后不可改地址`、`先安抚再处理`

后续评估时，可以将规则准确率定义为：

- 模型回复是否符合 `platform_rule`
- 模型回复是否与 `policy_tags` 对应的业务策略一致

## 8. Difficulty 设计

当前样本难度分为：

- `easy`
- `medium`
- `hard`

设计目的：

- `easy`
  - 单一规则判断，处理路径清晰。
- `medium`
  - 包含规则边界、状态差异或拒答约束。
- `hard`
  - 涉及投诉情绪安抚、复杂售后、不能只道歉不处理等情况。

后续可以用该字段做：

- 分难度规则准确率统计
- 分难度胜率对比
- badcase 集中区分析

## 9. 数据生成流程

当前数据生成流程已封装在 [data_scripe.py](/Users/dongyong/Project/Trea_code/ecom-llm-cs/data_scripe.py) 中，整体分为以下步骤：

1. 预定义客服场景与规则模板
2. 随机采样商品类别、订单状态、用户情绪等上下文变量
3. 根据场景与规则构建结构化样本骨架
4. 调用生成模型补全高质量客服回复
5. 用 schema 校验函数验证字段完整性
6. 输出标准化 JSON 数据集

该流程的目标不是机械扩写，而是生成具有业务条件、规则标签和难度标记的训练样本。

## 10. 数据设计的项目价值

这套数据设计的价值主要体现在以下几点：

- 提升训练输入的一致性与信息密度
- 支持规则准确率而不是只看文本相似度
- 支持按场景、标签、难度做细粒度分析
- 支持 LoRA 消融实验中的控制变量比较
- 支持从 badcase 反向定位具体规则缺陷

相比仅构造简单问答对，结构化 schema 更接近真实工业级大模型项目中的数据工程思路。

## 11. 后续优化方向

数据设计后续可以继续从以下方向优化：

- 增加更多一级和二级场景
- 扩展更多业务状态槽位，例如是否过保、是否定制商品、是否使用优惠券
- 引入人工审核样本，提升高价值 eval set 质量
- 对 `policy_tags` 进行更细粒度拆分
- 引入多轮对话字段，为后续对话式客服任务做准备

## 12. 小结

本项目的数据设计已经从单纯的问答对，升级为面向训练、评估和 badcase 分析的统一结构化 schema。该设计为后续 LoRA 消融实验、规则准确率统计和项目面试表达提供了稳定基础。
